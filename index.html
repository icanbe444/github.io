Cyclistic Bike Sharing Analysis

May 2024

Overview

Cyclistic is a fictional bike-share company. The objective was to understand behavioural differences between casual riders and annual members in order to support a marketing strategy aimed at converting casual riders into subscribers. I took the project from raw data through to a polished executive presentation.

What I Did

Collected and cleaned 12 months of trip data, resolving null values, duplicate records and inconsistent column formats. Built a series of visualizations that clearly showed where and when casual riders diverge from members — arming the marketing team with specific, actionable targeting opportunities.

Key Outcomes

  • Identified that casual riders average 2× longer trips than members, concentrated on weekends
  • Pinpointed the top 10 stations used almost exclusively by casual riders — direct marketing targets
  • Delivered a fully annotated presentation ready for stakeholder review

Data Warehouse Project

Oct 2025

Overview

Designed and built a SQL-based data warehouse from scratch to consolidate data from multiple source systems into a single, query-optimised analytical store. The project followed a layered architecture — bronze (raw ingestion), silver (cleaned and conformed), gold (aggregated, business-ready) — to ensure clean separation of concerns and maintainability.

What I Did

Defined the warehouse schema using a star schema model with clearly separated fact and dimension tables. Wrote ETL scripts to extract, transform and load data from source CSVs, handling type coercion, null management and deduplication at the silver layer. Created documentation covering table definitions, lineage diagrams, and loading procedures so the warehouse could be maintained and extended by others.

Key Outcomes

  • Reduced average analytical query time by structuring data for read-optimised joins
  • Produced a fully documented schema enabling team onboarding without tribal knowledge
  • Established a repeatable ETL pattern reusable across future data ingestion projects

End-to-End Azure Data Engineering Pipeline

2025

Overview

Designed and built a complete cloud-based data engineering pipeline on Microsoft Azure — from raw HTTP data ingestion through to a Power BI reporting layer. The project follows a medallion architecture (bronze → silver → gold), using Azure-native services at every stage to ensure scalability, security and maintainability.

Architecture

Data is ingested from an HTTP source via Azure Data Factory into a Data Lake Gen2 raw store (bronze layer). Azure Databricks handles all transformation logic, cleaning and conforming data into the silver layer before writing to a second Data Lake Gen2 store. The transformed data is then served via Azure Synapse Analytics (gold layer), which feeds directly into Power BI for reporting. Access control is managed through Microsoft Entra ID.

What I Did

Configured the end-to-end pipeline from scratch — provisioning all Azure services, designing the Data Factory pipelines, writing Databricks notebooks for transformation logic, and setting up Synapse as the serving layer. Implemented role-based access control via Entra ID across all components and documented the full architecture for handover and reproducibility.

Key Outcomes

  • Delivered a fully automated, end-to-end pipeline requiring no manual data movement
  • Medallion architecture ensures clean separation between raw, clean and business-ready data
  • Power BI report layer connects directly to Synapse for real-time, query-on-demand reporting
  • Full documentation produced covering architecture decisions, setup steps and access model

SQL Data Cleaning Project

Sep 2024

Overview

Designed and implemented a comprehensive SQL-based data cleaning pipeline to transform a raw, messy dataset into a reliable, analysis-ready form. The project established a set of reusable data quality checks and transformation patterns that could be applied across future datasets — not just a one-off fix, but a scalable framework.

What I Did

Audited the raw dataset systematically to catalogue data quality issues: nulls, duplicates, inconsistent formatting, out-of-range values and orphaned records. Wrote modular SQL scripts addressing each category of issue, with inline comments explaining the rationale for each transformation. Documented the full process in a data quality log that records what was changed, why, and what the source data looked like before and after.

Key Outcomes

  • Reduced null and duplicate records from ~18% of rows to under 0.5%
  • Standardised date and string formats across 14 columns for consistent downstream querying
  • Produced a documented, repeatable cleaning framework reusable on future projects
